## Entry Points

- `payment_gateway/adapters.py` - `parse_transaction()` function
- `billing_service/handlers.py` - `parse_transaction_response()` function  
- `reconciler/core.py` - `parse_txn()` function
- `audit_logger/extract.py` - `extract_transaction()` function
- `client.py` - `post_payment()` function
- `worker.py` - `run_with_retry()` function
- `scheduler.py` - `schedule_job()` function

## Data Flow

1. Transaction data flows through parsing functions in multiple modules (`payment_gateway/adapters.py`, `billing_service/handlers.py`, `reconciler/core.py`, `audit_logger/extract.py`)
2. Configuration values are imported and used across modules (`config.py` - TIMEOUT_SECONDS, MAX_RETRIES, JOB_INTERVAL_SECONDS)
3. HTTP requests are made via `client.py`'s `post_payment()` function
4. Retry logic is implemented in `worker.py` using `run_with_retry()`
5. Scheduling is handled by `scheduler.py`'s `schedule_job()`

## Key Modules

- **Parsing modules**: `payment_gateway/adapters.py`, `billing_service/handlers.py`, `reconciler/core.py`, `audit_logger/extract.py` - all contain identical transaction parsing logic
- **Configuration**: `config.py` - centralizes timeout, retry, and scheduling constants
- **Client interaction**: `client.py` - HTTP request handling with timeout configuration
- **Retry mechanism**: `worker.py` - retry wrapper function using MAX_RETRIES
- **Scheduling**: `scheduler.py` - job scheduling with interval configuration
- **Service modules**: 452 modules (`service_000/module_000.py` through `service_451/module_451.py`) - each contains a service class with similar structure

## External Dependencies

- `requests` library (used in `client.py`)
- Configuration constants from `config.py`
- Test module `tests/test_scheduler.py` that references `JOB_INTERVAL_SECONDS` and `TIMEOUT_SECONDS`

## Hidden Coupling

1. **Identical parsing logic**: All four transaction parsing functions (`parse_transaction`, `parse_transaction_response`, `parse_txn`, `extract_transaction`) are exact duplicates with identical implementation
2. **Configuration coupling**: Multiple modules import and use the same configuration constants from `config.py`
3. **Test dependency**: Test file `tests/test_scheduler.py` directly references module-level constants, creating tight coupling between test and implementation
4. **Service class duplication**: 452 service classes all follow identical pattern with only naming differences (service_000 through service_451)
5. **Module naming convention**: All service modules follow predictable naming pattern (`service_XXX/module_XXX.py`)

## Risky Files

- `payment_gateway/adapters.py`, `billing_service/handlers.py`, `reconciler/core.py`, `audit_logger/extract.py` - contain identical code that should be consolidated
- `config.py` - central configuration file that affects multiple modules; changes could have wide impact
- `tests/test_scheduler.py` - test directly references module constants, making tests brittle to implementation changes
- All 452 service modules (`service_XXX/module_XXX.py`) - massive duplication with no clear benefit

## Suggested Refactor Areas

1. **Consolidate parsing functions**: Extract identical transaction parsing logic into a single shared utility module
2. **Reduce service class duplication**: Replace 452 similar service classes with a single configurable service class or factory pattern
3. **Centralize configuration management**: Consider using a configuration manager class instead of direct imports from `config.py`
4. **Improve test isolation**: Refactor `tests/test_scheduler.py` to not directly reference module constants
5. **Module naming consistency**: Standardize the naming and structure of service modules to avoid 452 similar files
6. **Extract common patterns**: Identify and extract common patterns from service classes into base classes or mixins
